[CRIT-01] GET /settings expone tokens OAuth sin autenticación
--------------------------------------------------------------------
Archivo: backend/src/modules/settings/settings.controller.ts (línea 77-79)
         backend/src/modules/settings/settings.service.ts (línea 16-43)

Funcionamiento actual:
  El endpoint GET /api/settings no tiene ningún guard de autenticación
  (@UseGuards). Cualquier persona, sin login, puede hacer una petición
  GET y recibir TODA la configuración del sistema. El método get() del
  servicio hace un spread completo del registro Setting de Prisma
  (...settings), lo que incluye campos sensibles como:
  - mpAccessToken  (token de acceso OAuth de Mercado Pago)
  - mpRefreshToken (token para renovar el access token)
  - mpTokenExpiresAt
  - mpCollectorId  (ID de cuenta de Mercado Pago)
  - mpStoreId / mpPosId (IDs de tienda y punto de venta)
  - mpQrData       (datos del QR del POS)

Riesgo:
  Un atacante puede obtener tokens OAuth válidos y operar la cuenta de
  Mercado Pago vinculada. Puede crear órdenes de pago, leer transacciones,
  y potencialmente desviar pagos.

Solución propuesta:
  - Agregar @UseGuards(JwtAuthGuard) al método GET.
  - En el servicio, excluir explícitamente los campos sensibles
    (mpAccessToken, mpRefreshToken) del objeto de respuesta con un select
    o delete.


[CRIT-02] Tokens OAuth se loguean completos en mensajes de error
--------------------------------------------------------------------
Archivo: backend/src/modules/mercadopago-oauth/mercadopago-oauth.service.ts (línea 114)
         backend/src/modules/common/mp-config.service.ts (línea 145)

Funcionamiento actual:
  Cuando falla una llamada a la API de Mercado Pago (intercambio de código
  OAuth o refresh de token), el código actual loguea el objeto de respuesta
  COMPLETO usando JSON.stringify():
  - this.logger.error(`MP OAuth token exchange failed: ${JSON.stringify(tokenData)}`)
  - this.logger.error(`MP OAuth refresh failed: ${JSON.stringify(data)}`)
  El objeto contiene access_token y refresh_token en texto plano.

Riesgo:
  Los logs de contenedor suelen enviarse a sistemas centralizados (ELK,
  CloudWatch, Datadog, etc.). Los tokens quedan expuestos en esos sistemas
  y cualquiera con acceso a los logs puede extraerlos.

Solución propuesta:
  - Antes de loguear, crear una copia del objeto y eliminar/redactar los
    campos access_token y refresh_token. Ejemplo:
    const { access_token, refresh_token, ...safe } = tokenData;
    this.logger.error(`MP error: ${JSON.stringify(safe)}`);

[ALTA-01] CORS hace fallback a wildcard "*"
--------------------------------------------------------------------
Archivo: backend/src/main.ts (línea 15)

Funcionamiento actual:
  const corsOrigin = config.get<string>('CORS_ORIGIN') || '*';
  app.enableCors({ origin: corsOrigin });
  Si la variable de entorno CORS_ORIGIN no está definida, se usa '*' como
  origen permitido. Esto significa que cualquier dominio puede hacer
  peticiones AJAX al backend. Tampoco se restringen métodos HTTP ni headers.

Riesgo:
  Cualquier sitio web malicioso puede hacer peticiones autenticadas al
  backend si el usuario tiene una sesión activa (el navegador envía
  cookies y tokens automáticamente en peticiones cross-origin sin
  credenciales restringidas).

Solución propuesta:
  - Eliminar el fallback || '*'. Hacer que CORS_ORIGIN sea obligatorio.
  - Si no está definido, lanzar un error en startup.
  - Agregar restricción de methods: ['GET','POST','PATCH','DELETE'] y
    allowedHeaders explícitos.


[ALTA-02] Sin rate limiting en ningún endpoint
--------------------------------------------------------------------
Archivo: No implementado en todo el backend

Funcionamiento actual:
  El proyecto no tiene instalado @nestjs/throttler ni ningún mecanismo
  de rate limiting. Todos los endpoints aceptan requests ilimitados sin
  ningún tipo de control de frecuencia, incluyendo:
  - POST /auth/login (sin límite de intentos)
  - GET /auth/login-users (endpoint sin auth que expone usuarios)
  - POST /webhooks/mercadopago (potencial DoS)

Riesgo:
  - Fuerza bruta contra login: un atacante puede probar miles de
    contraseñas por minuto sin ser bloqueado.
  - Enumeración de usuarios vía /auth/login-users sin restricción.
  - Denegación de servicio por saturación de endpoints.

Solución propuesta:
  - Instalar @nestjs/throttler.
  - Configurar límites por IP: 5 intentos/min en /auth/login,
    30 req/min en endpoints generales.
  - Opcional: implementar bloqueo progresivo de cuenta tras N fallos.


[ALTA-03] Sin security headers (Helmet)
--------------------------------------------------------------------
Archivo: backend/package.json (helmet no instalado)
         backend/src/main.ts (no se configura helmet)

Funcionamiento actual:
  El proyecto no usa el paquete helmet ni configura manualmente headers
  de seguridad HTTP. Faltan headers como:
  - X-Content-Type-Options: nosniff
  - X-Frame-Options: DENY
  - Strict-Transport-Security (HSTS)
  - Content-Security-Policy
  - Referrer-Policy
  - Permissions-Policy

Riesgo:
  - Clickjacking: el sitio puede ser embebido en un iframe malicioso.
  - MIME sniffing: el navegador puede interpretar archivos como otro
    tipo de contenido.
  - Sin HSTS, conexiones HTTP sin protección contra downgrade.
  - Sin CSP, mayor superficie de ataque XSS.

Solución propuesta:
  - Instalar helmet: npm install helmet
  - En main.ts: app.use(helmet()) antes de cualquier otra configuración.

[ALTA-05] Webhook de MercadoPago loguea request completo
--------------------------------------------------------------------
Archivo: backend/src/modules/sales/webhooks/mercadopago-webhook.controller.ts (línea 24-32)

Funcionamiento actual:
  Cuando llega un webhook de Mercado Pago, el controlador loguea el
  request entero con JSON.stringify:
  {
    method, url, query,
    headers,  // incluye x-signature, x-request-id
    body      // incluye datos de pago, resource, topic
  }
  Esto se ejecuta en cada webhook recibido, antes de cualquier validación.

Riesgo:
  - El header x-signature queda registrado en logs (aunque no es

  - Si los logs se almacenan en servicios externos, datos de clientes
    quedan expuestos.

Solución propuesta:
  - Loguear solo campos esenciales: topic, resourceId, requestId.
  - No loguear headers completos ni body completo.
  - Usar nivel debug para datos detallados y solo en development.


[ALTA-06] Webhook signature bypass para topic merchant_order
--------------------------------------------------------------------
Archivo: backend/src/modules/sales/webhooks/mercadopago-webhook.utils.ts (línea 106-110)
         backend/src/modules/sales/webhooks/mercadopago-webhook.controller.ts

Funcionamiento actual:
  En la verificación de firma del webhook, si el topic es "merchant_order",
  aunque la firma sea inválida, el sistema devuelve { shouldReject: false }
  y continúa procesando. Esto significa que cualquier persona que conozca

  etc.).

Solución propuesta:
  - Tratar merchant_order igual que payment: validar firma siempre.
  - Si MP_WEBHOOK_STRICT_PAYMENT=true, rechazar órdenes sin firma válida.
  - Como mínimo, loguear una alerta de seguridad cuando se recibe un
    merchant_order con firma inválida.


[ALTA-07] Bypass de firma en entornos no-producción
--------------------------------------------------------------------
Archivo: backend/src/modules/sales/webhooks/mercadopago-webhook.utils.ts (línea 88-90)

Funcionamiento actual:
  Si NODE_ENV !== 'production' y no hay MP_WEBHOOK_SECRET configurado,
  la verificación de firma se saltea completamente y retorna { isValid: true }.
  Esto permite que cualquier request al webhook sea aceptado como válido
  en entornos de development/staging.

Riesgo:
  Si una instancia se despliega accidentalmente con NODE_ENV=development
  (o simplemente no se configura NODE_ENV), los webhooks no se verifican.

  arbitrarios que serán procesados como legítimos.

Solución propuesta:
  - Siempre verificar la firma si MP_WEBHOOK_SECRET está configurado,
    independientemente del entorno.
  - Solo saltear la verificación si no hay secreto Y es development,
    y en ese caso loguear una advertencia muy visible.

[MED-01] JWT almacenado en localStorage (vulnerable a XSS)
--------------------------------------------------------------------
Archivo: frontend/src/api/client.ts (línea 11)
         frontend/src/context/AuthContext.tsx (línea 20, 30)

Funcionamiento actual:
  Al hacer login, el backend devuelve { accessToken } en el body JSON.
  El frontend guarda este token en localStorage.getItem/setItem('authToken').
  En cada request, el interceptor de Axios lee el token de localStorage y
  lo agrega al header Authorization: Bearer <token>.

Riesgo:
  Cualquier script JavaScript que se ejecute en el contexto de la página
  (por ejemplo, un ataque XSS o una extensión maliciosa del navegador)
  puede leer localStorage y robar el token JWT. A diferencia de las
  cookies httpOnly, localStorage es accesible desde JavaScript.

Solución propuesta:
  - Ideal: usar cookies httpOnly + secure + sameSite para el token.
  - Alternativa más simple: mantener localStorage pero agregar una capa
    de refresh token con rotación y expiración corta (15-30 min).
  - Implementar Content-Security-Policy estricta para mitigar XSS.


[MED-02] Contraseñas débiles (mínimo 6 caracteres)
--------------------------------------------------------------------
Archivo: backend/src/modules/users/dto/create-user.dto.ts (línea 14)
         backend/src/modules/auth/dto/login.dto.ts

Funcionamiento actual:
  El DTO de creación de usuario y el de login usan @MinLength(6) como
  única restricción de contraseña. No se requiere mayúsculas, números,
  ni caracteres especiales. El login también acepta un campo "pin"
  numérico de mínimo 6 dígitos como alternativa a la contraseña.

Riesgo:
  Contraseñas de 6 caracteres sin complejidad son vulnerables a fuerza
  bruta y ataques de diccionario. Un PIN de 6 dígitos tiene solo 10^6
  combinaciones (1 millón), lo cual es trivial para un ataque automatizado
  sin rate limiting.

Solución propuesta:
  - Aumentar mínimo a 8 caracteres.
  - Agregar requisitos de complejidad: al menos 1 mayúscula, 1 número.
  - Para PIN: mínimo 6 dígitos está bien si hay rate limiting, pero
    considerar aumentar a 8 dígitos o bloquear tras N intentos.

[MED-05] Extensión de archivo controlada por usuario en uploads
--------------------------------------------------------------------
Archivo: backend/src/modules/settings/settings.controller.ts (línea 31-33)

Funcionamiento actual:
  Los uploads de logo y favicon usan diskStorage. El nombre del archivo
  se genera como:
    `${Date.now()}-${Math.random()}${extname(file.originalname)}`
  La extensión se toma de file.originalname (controlado por el usuario)
  con extname(). Aunque hay validación de mimetype (image/*), un archivo
  con mimetype image/svg+xml y extensión .svg se guardaría como SVG.
  El SVG puede contener JavaScript embebido que se ejecutaría si el
  navegador lo carga como imagen en un contexto que permita scripts.

Riesgo:
  Stored XSS mediante SVG malicioso subido como logo. Si el SVG se sirve
  con Content-Type: image/svg+xml, los scripts embebidos se ejecutan.

Solución propuesta:
  - No usar extname(file.originalname). Generar la extensión basada en
    el mimetype validado (ej: image/png → .png).
  - Procesar el logo con Sharp y convertirlo a PNG/WebP siempre,
    eliminando cualquier posibilidad de SVG malicioso.
  - Agregar Content-Security-Policy que bloquee scripts en imágenes.

[MED-07] Sin DTO validation en endpoints de OAuth
--------------------------------------------------------------------
Archivo: backend/src/modules/mercadopago-oauth/mercadopago-oauth.controller.ts (líneas 20, 40, 45-54)

Funcionamiento actual:
  Los endpoints del controlador de OAuth aceptan datos con tipos inline
  de TypeScript en vez de usar DTOs con class-validator:
  - POST /mp-oauth/token: @Body() body: { code: string }
  - POST /mp-oauth/select-store: @Body() body: { storeId: string; posId: string }
  - POST /mp-oauth/setup-pos: 8 campos string sin validación

Riesgo:

  adicionales que no deberían aceptarse.

Solución propuesta:
  - Crear DTOs: TokenExchangeDto, SelectStoreDto, SetupPosDto.
  - Agregar decoradores de class-validator (@IsString, @IsNotEmpty, etc.)
    a cada campo.
  - Usar los DTOs en los parámetros @Body().


[MED-08] Condición de carrera en decremento de stock
--------------------------------------------------------------------
Archivo: backend/src/modules/sales/sales.service.ts (líneas 565-585)

Funcionamiento actual:
  Al registrar una venta con productos que tienen recetas (ingredientes),
  el sistema primero lee el stock actual con findUnique y luego actualiza
  con $executeRaw:
    1. SELECT stock FROM "Product" WHERE id = ...
    2. UPDATE "Product" SET stock = ${nuevo_stock} WHERE id = ...
  Entre el SELECT y el UPDATE, otra venta concurrente puede leer el mismo
  stock. Ambas calcularán el nuevo stock basándose en el valor original,
  causando que uno de los decrementos se "pierda".

Riesgo:
  El stock puede quedar inconsistente (más alto de lo real) bajo carga
  concurrente. Esto puede llevar a vender productos sin stock real.

Solución propuesta:
  - Usar Prisma update con decrement atómico:
    prisma.product.update({
      where: { id },
      data: { stock: { decrement: cantidad } }
    })
  - O usar transacción con bloqueo pesimista (SELECT ... FOR UPDATE).


[MED-09] Items de ConfirmTransfer sin validación anidada
--------------------------------------------------------------------
Archivo: backend/src/modules/payments/payments.controller.ts
         backend/src/modules/payments/dto/transfer.dto.ts

Funcionamiento actual:

  ConfirmTransferWithItemsDto que extiende ConfirmTransferDto pero agrega

  No se incluyen claims estándar como iss (issuer) ni aud (audience).

Riesgo:
  Muy bajo. Son claims recomendados para prevenir el uso de tokens en
  sistemas incorrectos. En una arquitectura simple como esta, el riesgo
  es mínimo.

Solución propuesta:
  - Agregar iss: 'm-posw-api' y aud: 'm-posw-client' al payload.
  - Validarlos en JwtStrategy si se agregan.


[BAJ-09] JWT sin algoritmo explícito en la estrategia
--------------------------------------------------------------------
Archivo: backend/src/modules/auth/jwt.strategy.ts

Funcionamiento actual:
  La estrategia JWT de Passport no especifica el campo algorithms en
  sus opciones. Usa el default (HS256). Si bien Passport-jwt maneja
  correctamente el algoritmo, es buena práctica ser explícito.

Riesgo:
  Muy bajo. Passport-jwt extrae el algoritmo del token decodificado
  y lo valida contra la clave. La ausencia de configuración explícita
  no debería causar problemas con versiones actuales.

Solución propuesta:
  - Agregar algorithms: ['HS256'] en las opciones de JwtStrategy.
